The question that kept me up at night was never about databases or scaling. It was about shape. What is the interface that everything connects to?

In 2017, I sat across from Jamie McCrindle and Johnny Reilly in an Investec interview room and told them I had the answer: "DynamoDB + strong types + GraphQL = universal interface = infinitely scalable app."

They were not impressed. They challenged it politely, and I defended my position with the confidence of someone who had never been proven wrong because they had never tested the idea at real scale. Deep down, a seed of doubt was planted. Nine years later, having migrated systems from Next.js to AWS to Cloudflare Workers, I finally understand what they sensed that I could not yet see.

## Why GraphQL Felt Universal

GraphQL checked every box on paper. Type safety. A single endpoint. Introspection. A schema-first approach that let you query exactly what you needed. It felt like the last interface you would ever need — the way a well-organized filing cabinet feels like the answer to information management.

But it missed the point entirely.

GraphQL is a data layer. A way for machines to format data for other machines. It solves the question "how do I get exactly the right fields?" It does not solve the question "how do two entities — human or machine — work together on something?"

It is the difference between a library catalogue and a conversation. The catalogue is perfect for finding a book. It is useless for figuring out which book you should read.

## The AWS Migration and the Complexity Tax

As our platforms grew, we migrated from Next.js to AWS, chasing flexibility. The cost improved initially, but the complexity exploded like a gas expanding to fill its container.

Lambda cold starts. DynamoDB capacity planning. Intricate CloudFormation templates spanning three regions. Nearly two years spent on pure infrastructure work before we could ship a single new feature.

The "infinitely scalable" dream came with an exorbitant complexity tax. We were spending more time orchestrating infrastructure than building the things that users actually touched. The system scaled beautifully. The team did not.

## The Cloudflare Pivot

Then came Cloudflare Workers. We moved everything to the edge. The change was profound: zero cold starts, global distribution by default, D1 for data, Durable Objects for real-time state.

Costs dropped. But the real win was not cost — it was simplicity. The platform became something a small team could hold in their heads again. The number of moving parts collapsed. The feedback loop between "I changed something" and "I can see the result" went from minutes to seconds.

Simplicity is not the absence of capability. It is capability with fewer joints that can break.

## What Jamie and Johnny Were Right About

Three migrations. Nine years. A Cloudflare Workers bill that is a fraction of the AWS bill. And the question they asked in that interview is still the right one.

Looking back, I realize what Jamie and Johnny sensed: GraphQL was not wrong — it was answering a question that was not the important one. They could not have told me what the actual universal interface was. Nobody could have in 2017. AI agents and the Model Context Protocol did not exist yet. But their skepticism was well-placed. A query language solves data retrieval. It does not solve collaboration.

Nine years later, I think I know the answer. The universal interface is the one humans have used since before writing existed: conversation. Chat. The back-and-forth of two entities working through a problem together.

When you add MCP into the mix, you get a tool layer that lets those conversations drive real actions — not just exchange information, but actually do things.

Together, **Chat + MCP = the actual universal interface.**

## Introducing spike-chat

This realization led directly to building `spike-chat`, a standalone Cloudflare Worker that provides real-time messaging for the entire spike.land platform.

The architecture is built on the same primitives that made our platform simple again:
- **Channels, threads, and DMs** all natively supported via D1 and Durable Objects.
- **Agent-as-first-class-citizen:** AI agents operate within the chat seamlessly, responding to events and executing tools. They are not bolted on — they are participants.
- Any app on our platform can enable this chat with a single line of configuration.

Here is a live demo. We have embedded a public channel right here in the blog post. Anyone, including visitors, can post (with rate-limited guest access). Try it out and say hello.

```tsx
import { SpikeChatEmbed } from "@spike-land/core/block-website/ui/SpikeChatEmbed";


<SpikeChatEmbed
  channelSlug="blog-universal-interface"
  workspaceSlug="spike-land"
  guestAccess={true}
  height={500}
/>
```

## The DX That Actually Impresses

Developer experience is the quiet force multiplier. Instead of setting up OAuth apps, managing scopes and tokens, configuring event subscriptions — the usual ceremony that takes a week before you can send your first message — enabling chat in a workspace app looks like this:

```json
{
  "chat": {
    "enabled": true
  }
}
```

That is it. On the frontend, the planned API for connecting to a channel is a single React hook:

```tsx
const { messages, sendMessage } = useChatMessages({ channelId: "general" });
```

It is the simplicity we were always chasing but never finding with a query language. GraphQL answered "how do I get data?" Chat + MCP answers "how do we work together?"

Jamie, Johnny — I finally have the answer to your question. It took three migrations and nine years. And this time, the interface really is universal.
