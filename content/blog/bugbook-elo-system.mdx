We run 80+ MCP tools across half a dozen Cloudflare Workers. Previously, error logging was less structured, making insights harder to gather. Empowering users to contribute to platform improvement was a key goal. And our previous rate limiting system, while functional, didn't differentiate between new visitors and loyal power users.

Three separate challenges. One system addressed all of them.

Here is how.

---

## The Problem: Internal DDoS Prevention

When you expose 80 tools to the internet via MCP, maintaining platform stability and resource efficiency is paramount. A single agent in a retry loop could inadvertently stress our infrastructure — and from the outside, it looks identical to a legitimate user making rapid tool calls.

While IP-based rate limiting offered a baseline, we sought a more nuanced approach. Agents share IPs. VPNs exist. And the real question is not "how many requests per minute?" The real question is "should this caller be trusted at all?"

We needed something smarter. Not a wall, but a gradient — a system that adjusts itself based on behavior, the way a thermostat adjusts based on temperature.

We needed a reputation system. For more on building robust online trust, check out this insightful article on [Building Trust: The Science of Online Reputation Systems](https://medium.com/@jason.d.carter/building-trust-the-science-of-online-reputation-systems-30f14842142e).

## Why ELO?

We already had an ELO engine in our codebase — the chess package uses it for player rankings. ELO has a beautiful property that most ranking systems lack: it is a single number that encodes trust, and it adjusts itself continuously based on outcomes. For a deeper dive into ELO implementation, explore [How to Implement the Elo Rating System in Python](https://www.freecodecamp.org/news/elo-rating-system-implementation-in-python/).

In systems thinking, ELO is a stock with balancing feedback loops. Good behavior fills the tank. Positive engagement enhances it. The system self-corrects without anyone making manual decisions.

The insight was: **ELO is not just for chess players. It works for anything that competes for limited resources.**

So we gave ELO ratings to two things:
1.  **Bugs** — which bugs matter most?
2.  **Users** — who should be trusted with platform access?

## Bug-vs-Bug ELO

Every bug starts at ELO 1200. When someone reports a bug, that bug "wins" a match against a random bug in the same category. Its ELO goes up; the other's goes down.

This creates emergent prioritization:
- Frequently reported bugs climb the leaderboard naturally
- Old bugs gracefully recede in priority if not re-encountered.
- The ELO leaderboard becomes a live priority list that maintains itself

Our aim was to empower the system to automatically prioritize bugs. We wanted the bugs to tell us how important they are — through the collective signal of user reports. The priority is not assigned. It emerges.

### Bug Lifecycle

```
CANDIDATE → ACTIVE → FIXED → DEPRECATED
```

A bug starts as a CANDIDATE when first reported. After 3 independent reports, it auto-promotes to ACTIVE. When fixed, it is marked FIXED. If it is irrelevant for 5+ sessions, ELO decay pushes it toward DEPRECATED. The lifecycle runs itself.

## User ELO: Trust as a Number

Every user (and their agents) starts at ELO 1200. Actions shift the score:

| Event | ELO Change |
|-------|-----------|
| Report a valid bug | +25 |
| Bug you reported gets confirmed | +10 |
| Successful tool use | +1 |
| Misidentified bug report | -15 |
| Accessed rate limit threshold | -5 |
| Community moderation flag | -50 |

There is a daily gain cap of +100 to prevent gaming. The math is adapted from chess K-factors — new users have higher volatility (K=40) that stabilizes over time (K=16 above ELO 2400). Just like in chess, your early games matter more.

### Tier Gating

ELO maps to three tiers:

| Tier | ELO Range | Effect |
|------|-----------|--------|
| Free | 0-999 | 4x rate limit multiplier |
| Pro | 1000-1499 | Standard access |
| Elite | 1500+ | Full access, lower limits |

A brand-new user (1200 ELO) starts as Pro. Good behavior keeps them there. Actions inconsistent with positive engagement may lead to Free tier, where rate limits are 4x stricter — effectively 30 requests per minute instead of 120.

This is the anti-DDoS mechanism, and it is deliberately gentle. Users exhibiting patterns inconsistent with positive engagement experience adjusted service levels, encouraging positive behavior, while legitimate users never notice anything at all. It is a gradient, not a gate. And unlike a hard ban, recovery is always possible — just behave well and the score recovers.

### Tools Can Require Tiers

Any MCP tool can declare a `requiredTier`:

```typescript
{
  name: "expensive_operation",
  requiredTier: "elite",
  // ...
}
```

Most tools have no tier requirement. But computationally expensive operations — image generation, code compilation — can require Pro or Elite access. This ensures that computationally intensive operations are reserved for users who have established a positive reputation. Trust is earned, not assumed.

## Feedback Tools Everywhere

Every MCP server in our platform now has a feedback tool:

| Server | Tool Name |
|--------|-----------|
| spike-land-mcp | `mcp_feedback` |
| mcp-image-studio | `img_feedback` |
| hackernews-mcp | `hackernews_feedback` |
| esbuild-wasm-mcp | `esbuild_feedback` |
| openclaw-mcp | `openclaw_feedback` |

When an agent encounters an error, it can report it through the closest feedback tool. The report flows to the central Bugbook in spike-edge, where it is matched against existing bugs (by error code + service name) or creates a new entry.

For Cloudflare Workers, this uses service bindings — zero-latency, zero-cost inter-Worker communication. For Node.js MCP servers, it is an HTTPS POST to the edge API. For more on advanced serverless patterns, explore [Advanced Cloudflare Workers Patterns for Modern Web Development](https://dev.to/cloudflare/advanced-cloudflare-workers-patterns-for-modern-web-development-4e78).

Here is the insight that surprised us: **agents are exceptional bug reporters.** They hit edge cases humans never would. They describe the exact tool call, parameters, and error message with perfect recall. And they report immediately, offering fresh context and preventing delays often associated with traditional reporting methods.

## Centralized Error Collection

Every unhandled error in spike-edge now gets logged to D1 via `waitUntil`:

```typescript
app.onError((err, c) => {
  c.executionCtx.waitUntil(
    c.env.DB.prepare(
      "INSERT INTO error_logs (...) VALUES (...)"
    ).bind(/* ... */).run()
  );
  return c.json({ error: "Internal Server Error" }, 500);
});
```

This replaces less structured `console.error()` outputs with structured, queryable error data. Service name, error code, stack trace, request metadata — all searchable. The Bugbook API can correlate user reports with actual errors, closing the loop between "a user says something is broken" and "here is the stack trace that proves it."

The loop is the whole point. Reports without evidence are complaints. Evidence without reports is noise. Together they become signal.

## The Public Bugbook

The Bugbook is public at `/bugbook`. Anyone can see:
- All active bugs, ranked by ELO
- Bug detail pages with report history and ELO timeline
- A leaderboard showing top bugs and top reporters

Users who report bugs can track their reports and see when bugs get fixed. This is deliberate transparency. If you report a bug, you should not have to wonder if anyone noticed. The system shows you.

Authenticated users can:
- Submit new bug reports
- Confirm existing bugs ("I have this bug too")
- View their own report history

## Blog Comments with ELO Consequences

We also added comments to all blog articles. Logged-in users can comment at any point in an article, and other users can upvote or downvote comments.

Here is where it gets interesting: if a comment accumulates a score of -10 or lower (overwhelmingly downvoted), the comment author receives a `community_moderation_flag` ELO event (-50 points). The community self-moderates. Positive community engagement is encouraged, and actions inconsistent with this, such as spam or abuse, are collectively identified, leading to adjustments in platform reputation.

This is another emergent property. We did not build a moderation team. We built a system where moderation emerges from collective behavior.

## Architecture Summary

```
                    ┌──────────────────┐
                    │   spike-app      │
                    │   /bugbook UI    │
                    └────────┬─────────┘
                             │ HTTPS
                    ┌────────▼─────────┐
                    │   spike-edge     │
                    │  Bugbook API     │
                    │  ELO Engine      │
                    │  Error Logs      │
                    │  Blog Comments   │
                    └──┬──────────┬────┘
          Service      │          │     HTTPS
          Binding      │          │
    ┌──────────────┐   │    ┌─────▼──────────┐
    │spike-land-mcp│◄──┘    │ Node.js MCP    │
    │ mcp_feedback │        │ servers         │
    │ ELO gating   │        │ *_feedback      │
    └──────────────┘        └────────────────┘
```

Everything runs on Cloudflare Workers + D1. No external databases, no Redis, no queues. The ELO engine is pure math — 40 lines adapted from our chess package. The entire system adds about 500 lines of business logic and 200 lines of SQL.

We've found that self-regulating, lean systems consistently outperform larger, maintenance-heavy alternatives.

## What We Learned

1.  **ELO is a universal trust primitive.** Any system where entities compete for relevance can use ELO. Bugs compete for attention. Users compete for trust. The math is the same — only the domain changes.

2.  **Agents are exceptional bug reporters.** They report immediately, with full context, and they hit edge cases that manual testing misses. They are the best sensors in the system.

3.  **Reputation-based rate limiting is gentler and smarter than IP blocking.** Instead of hard walls, users exhibiting patterns inconsistent with positive engagement experience adjusted service levels. They can recover by behaving well. It is a gradient, not a gate — and gradients are harder to game.

4.  **Public bug tracking builds trust.** When users can see that their reports lead to fixes, they report more bugs. When they can see the priority order via ELO, they understand why some bugs get fixed before others. Transparency creates a reinforcing loop.

5.  **Feedback tools should be everywhere.** If every MCP server has a feedback tool, then every agent interaction is a potential bug report. The coverage is automatic. You do not have to ask for feedback — the system collects it by existing.
