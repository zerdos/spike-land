# What CCU Revealed — and What spike-chat Already Had

## The Discovery
Someone reverse-engineered Claude Code's Mach-O binary and npm package. They found named subsystems: Iron Gate (fail-closed safety), Auto-Dream (background memory consolidation), Kairos (auto-brief + cron scheduling), a two-stage classifier (fast + thinking), and 79 feature flags prefixed "tengu_".

## What spike-chat already had
- Scored memory notes with Bayesian confidence (their version: session memory, currently disabled)
- Four-stage pipeline with context isolation (their version: single-pass with feature-flagged capabilities)
- PRD compression filter (no equivalent found in CCU)
- Bounded tool calling with catalog hints (their version: tool surface gated by feature flags)

## What we learned and stole
1. **Iron Gate** — fail-closed classifier. If our classify stage fails, default to safe conversation mode. Never pass through unchecked.
2. **Two-stage classifier** — fast heuristic first, LLM second only when confidence is low. Saves latency on obvious intents.
3. **Auto-Dream** — background note consolidation. Merge duplicates, decay stale notes, promote consistent patterns. Runs idle, not per-request.
4. **Execution feedback loop** — tool success/failure feeds back into note confidence. The self-improving loop closes tighter.

## What we would not steal
- 79 feature flags (we ship, not flag)
- Session memory disabled by default (we ship memory on by default)
- Ultraplan remote multi-agent planning (hardcoded off — they are not sure about it either)

## The principle
The best AI systems are not the ones with the most hidden subsystems. They are the ones where every subsystem earns its place in the working set.

</Tldr>

## The Uncloaking

In late March 2026, a security researcher published [Claude Code Uncloaked](https://ccu.galdoron.com/#systems) — a full teardown of Claude Code's native binary and JavaScript internals.

They pulled roughly 42,000 unique strings from the macOS arm64 binary. They AST-parsed the npm package. They mapped every feature flag, every hidden subsystem, every configuration path.

What they found was not a monolith. It was an ecosystem.

Think of it like opening the hood of a car and finding not one engine, but a dozen specialists — each doing one job, each wired to the others through clean interfaces:

| CCU System | What It Does |
|---|---|
| **Iron Gate** | Fail-closed safety for auto-mode. Blocks actions when classifier is unavailable. |
| **Auto-Dream** | Background memory consolidation. PID-locked. Runs on idle. |
| **Kairos** | Combined auto-brief and cron scheduling subsystem. |
| **Two-stage classifier** | Fast model + thinking model at temperature=0, 4096 token max. |
| **Willow Mode** | Idle detection after 75 min + 100K tokens. Surfaces context recap. |
| **Penguin Mode** | Fast-mode cooldown state machine with resetAt/reason metadata. |
| **Harbor** | MCP channel push notification with allowlist permissions. |
| **Teleport** | Session bundling for CLI-to-cloud transfer. |
| **Ultraplan** | Remote multi-agent planning. Hardcoded disabled. |

79 feature flags prefixed `tengu_`. 118 settings across 12 categories. A multi-layer permission system with managed enterprise overrides.

This is serious systems engineering. Not a single clever trick — a whole ecology of interlocking parts.

When we read the analysis, two things happened at once:

1. We recognized patterns we had already built.
2. We saw the gaps we had been too close to notice.

## What spike-chat Already Had

Before CCU dropped, spike-chat's Aether pipeline already embodied several of the same architectural ideas — sometimes under different names, sometimes with different tradeoffs. The convergence was reassuring. The divergences were instructive.

### Scored memory (their "Session Memory" — disabled)

CCU revealed a session memory system inside Claude Code. It is currently turned off.

spike-chat ships Bayesian memory by default. Every note carries a confidence score, a help count, and recency data. Notes below the demotion threshold (0.3) get pruned. Notes that keep proving useful get promoted. Selection is greedy under a fixed 800-token budget.

We called it the note engine. They called it session memory. Same core idea: carry forward what earns its place, discard what does not.

The difference is simple but telling: ours is on. Theirs is off. A system that remembers is fundamentally different from a system that could remember but chooses not to.

### Four-stage pipeline (their single-pass with flags)

Claude Code processes requests in a single pass, using feature flags to gate different behaviors within that pass. Think of it like a single conveyor belt with switches that route items to different stations.

spike-chat splits every request into four explicit stages: classify, plan, execute, extract. Each stage receives the prior stage's compact artifact, not the full raw process. Think of it like separate rooms — each one gets a clean brief from the room before it, not the accumulated mess of everything that happened so far.

We split by design. They split by configuration. Both work. Ours is harder to accidentally break.

### PRD compression (no CCU equivalent found)

This is one area where spike-chat appears to be ahead.

CCU did not reveal a PRD compression layer. Claude Code manages context primarily through prompt caching, token-efficient tool use, and prompt engineering conventions.

spike-chat has a three-tier PRD filter: passthrough for short messages, template extraction for structured intents, and Gemini Flash compression for complex messages. Every chat message gets evaluated against this pipeline before it reaches the model.

The insight behind it: old conversations are latent product requirements. They contain decisions, constraints, preferences. We extract those. Most systems just replay the raw transcript.

### Bounded tool calling (their feature-flagged tool surface)

Claude Code gates tool availability through feature flags and permission modes.

spike-chat bounds the tool surface structurally: fetch the MCP catalog, limit to a compact set, hand it only to the execute stage. Tool definitions are wired into the model call, not just advertised by the route.

Same destination, different roads.

## What We Learned — and Stole

### 1. Iron Gate: Fail-Closed Classifier

This was the most important lesson from the entire CCU analysis.

Iron Gate is beautifully simple: if the auto-mode classifier is unavailable, block the action. Do not pass through. Do not guess. Fail closed.

It is the same principle as a dead man's switch on a train. If the driver lets go, the train stops. It does not speed up. It does not keep going at the same speed. It stops.

spike-chat's classify stage had no equivalent. If the heuristic classifier returned a plausible result, it was trusted. If something went wrong, the request still flowed through to execution.

That is fail-open. It is the wrong default for a system that can take real actions in the world.

We added an Iron Gate:

```typescript
// If classification confidence is below the gate threshold,
// force safe conversation mode — no tools, no browser actions.
if (intent.confidence < IRON_GATE_THRESHOLD) {
  return {
    intent: "conversation",
    urgency: "low",
    needsTools: false,
    confidence: intent.confidence,
    reason: "Iron gate: classification confidence below threshold.",
    gated: true,
  };
}
```

The rule: **if you are not confident about what the user wants, do less, not more.**

Tool use is powerful. Unconfident tool use is dangerous. The gate ensures that uncertainty defaults to safety — the system talks instead of acting.

### 2. Two-Stage Classifier: Fast Heuristic + LLM Confirmation

CCU's classifier uses two models: a fast one for initial classification, then a thinking model for confirmation. Both at temperature=0. Max 4096 tokens.

This is like a hospital triage system. The nurse at the front desk handles the obvious cases instantly — broken arm, go to orthopedics. But when someone walks in with vague symptoms, the nurse does not guess. She calls a doctor.

spike-chat's classifier was single-stage: regex heuristics that pattern-match against the user message. Fast, but brittle for anything ambiguous.

We kept the fast heuristic as stage one. But now, when confidence is low (but above the Iron Gate), we call in the doctor:

```text
User message
  → Stage 1: Regex heuristics (0-latency, high confidence on clear intents)
  → if confidence < 0.7 and confidence >= IRON_GATE_THRESHOLD:
      Stage 2: LLM classifier (temperature=0, max 256 tokens)
  → if confidence < IRON_GATE_THRESHOLD:
      Iron Gate: force conversation mode
```

"Fix the login bug" hits the fast path — obvious intent. "Can you check if the thing from yesterday is working" gets the second opinion — ambiguous, needs context. Garbage gets gated.

### 3. Auto-Dream: Background Note Consolidation

This is the one that excited us most, because it solves a problem we had been watching grow.

CCU's Auto-Dream is a background memory consolidation system. PID-locked. Runs on idle. Server-gated.

spike-chat's extract stage runs per-request, capturing one note per conversation turn. Good for learning new lessons. But it never tidied up.

Imagine a filing cabinet where you add one folder per day but never remove, merge, or reorganize anything. After six months, you have a filing cabinet full of duplicates, contradictions, and outdated information. The drawer still opens, but finding anything useful takes longer than it should.

Over time, spike-chat's note store accumulated exactly those problems:
- duplicate notes piling up
- conflicting notes coexisting peacefully
- stale notes lingering above the demotion threshold
- the token budget wasted on redundancy

Auto-Dream fixes this. We added a consolidation pass that runs after the extract stage:

- **Merge**: notes with similar triggers and lessons get consolidated into one higher-confidence note
- **Decay**: notes untouched for 30+ days get a confidence penalty
- **Promote**: notes that survived consolidation with high help counts get boosted
- **Prune**: notes that fall below the demotion threshold after decay get removed

This runs in the background via `waitUntil` — no latency added to the user's response.

The note engine was already Bayesian. Auto-Dream makes it self-cleaning. The filing cabinet now has a night janitor.

### 4. Execution Feedback Loop: Closing the Self-Improving Circle

This was the piece that had been implicit in spike-chat but never wired up. Like a thermostat that measures the temperature but never adjusts the heater.

The Aether pipeline has four stages: classify, plan, execute, extract. The extract stage learns from conversations. But the execute stage's tool outcomes — success, failure, timeout — were never fed back into note confidence.

That meant: if a note suggested an approach that consistently led to tool failures, the note kept its score. If a note's advice consistently led to successful executions, it got no reward.

The feedback loop was open. The system could learn new things but could not learn from its own results.

We closed it:

```typescript
// After tool execution, update notes that influenced this response
for (const note of selectedNotes) {
  const toolsSucceeded = toolOutcomes.every(o => o.status === "ok");
  updateNoteConfidence(note, toolsSucceeded);
}
```

Now the self-improving loop runs end to end:

1. User sends message
2. Relevant notes are selected from memory
3. Notes influence the classify, plan, execute pipeline
4. Tool execution outcomes are recorded
5. Extract stage learns new notes
6. **Execution outcomes update existing note confidence**
7. Auto-Dream consolidates in the background
8. Next message benefits from a cleaner, more accurate note set

Every note that helps earns trust. Every note that misleads loses it. The system gets sharper with use — not by remembering more, but by remembering better.

## What We Would Not Steal

### 79 feature flags

Claude Code uses 79 `tengu_`-prefixed feature flags for server-side capability gating.

We ship features or we do not. Flags are organizational tools for large teams coordinating across release trains. spike-chat is built by a solo founder with strong opinions. If a feature is ready, it ships. If it is not, it does not exist yet.

Every flag is a branch in the system's behavior that must be tested, documented, and eventually cleaned up. For a small team, the overhead of maintaining 79 possible configurations exceeds the benefit of being able to turn things on and off remotely. We would rather ship and iterate.

### Session memory disabled by default

CCU revealed that Claude Code's session memory is currently off.

spike-chat's Bayesian memory has been on since day one. It is load-bearing. Users notice when lessons they taught the system surface in later conversations.

We understand why Anthropic might keep it off — liability, privacy, enterprise compliance. Those are real concerns at their scale. But for spike-chat, memory is not a feature. Memory is the product. Turning it off would be like a restaurant removing the kitchen.

### Ultraplan remote multi-agent planning

CCU found this system hardcoded to disabled. 10-30 minute Opus planning on web. Multi-agent coordination.

We respect the ambition. But if the team that built Claude Code is not confident enough to enable it, that is a signal worth heeding. We will wait to see why before building our own version.

Single-agent with strong context discipline has been enough to rewrite an entire frontend from one PRD. We will add multi-agent when single-agent demonstrably fails — not before.

## The Erdős Observation

Paul Erdős had 511 co-authors because he believed the best mathematics emerges in the spaces between minds.

What CCU revealed is that Claude Code is not one mind. It is a collaboration of named subsystems — Iron Gate, Auto-Dream, Kairos, Willow, Harbor — each solving a specific problem, each with clear boundaries. A society of specialists, not a single generalist.

spike-chat's Aether pipeline was already built on the same principle: classify, plan, execute, extract. Each stage is a collaborator. Each stage receives the prior stage's artifact, not the full mess.

The CCU analysis validated the architecture. The gaps it revealed — fail-closed safety, two-stage classification, background consolidation, execution feedback — were real gaps. We closed them.

But the biggest lesson from CCU is not any single system.

It is the meta-lesson:

**The best AI products are systems of systems. And the quality of the whole depends on how honestly each subsystem earns its place.**

That is the Erdős principle applied to software: my brain is open, my budget is finite, and every note must earn its place in The Book.

## What Changed in spike-chat

| Before CCU | After CCU |
|---|---|
| Single-stage heuristic classifier | Two-stage classifier with confidence gate |
| Fail-open on classification uncertainty | Iron Gate: fail-closed to conversation mode |
| Per-request note extraction only | Auto-Dream background consolidation |
| Tool outcomes not fed back to memory | Execution feedback loop closes the circle |
| Notes accumulate without cleanup | Merge, decay, promote, prune cycle |

The self-improving loop is now:

```text
user message
  → select relevant notes (Bayesian, budget-bounded)
  → classify (heuristic → optional LLM → Iron Gate)
  → plan (artifact from classify)
  → execute (tools, bounded catalog, streaming)
  → extract (new notes from conversation)
  → feedback (tool outcomes → note confidence)
  → consolidate (Auto-Dream: merge, decay, promote, prune)
  → next message benefits from cleaner memory
```

Every turn makes the system sharper. Not by remembering more. By remembering better.

## The Falsifiable Claim

spike-chat's context discipline was already strong before CCU.

After these changes, the claim is sharper:

> spike-chat is one of the most disciplined context-engineered chat systems in production. Not because it has the most subsystems. Because every subsystem — from Iron Gate to Auto-Dream — earns its place in the working set.

That claim is falsifiable. Ship a transcript-first system with the same model and compare completion quality after 20 turns. We will take that bet.

Erdős would approve. The proof is constructive.
