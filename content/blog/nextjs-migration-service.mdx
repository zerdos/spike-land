## The Method: Six Steps, Same Order Every Time

This is not a methodology poster. This is the actual sequence we run. Every time. No shortcuts.

**1. Read the repo.** No destructive pipeline first. Inspect routing, detect framework imports, classify concerns, identify runtime dependencies. The migration equivalent of `bazdmeg:status`.

**2. Preview the transformation.** Before touching anything, generate a map: what moves, what gets rewritten, what stays, what looks suspicious. The migration equivalent of `bazdmeg:dry-run`.

**3. Separate mechanical from judgment.** This is the split that makes everything else possible. Route rewrites, config conversion, import replacement, directory reshaping — mechanical. Vercel platform assumptions, long-lived Node.js dependencies, custom SSR business logic — judgment. Hard repos become workable the moment you stop treating everything like custom engineering.

**4. Apply the deterministic pass.** Only after the preview matches intent. The boring parts become explicit and repeatable.

**5. Verify everything.** TypeScript must pass. Lint must pass. Tests must pass. Business flows must hold. If the diff cannot be explained by a human reviewer, it is not done.

**6. Escalate only what deserves human attention.** That is where expertise earns its value.

This sequencing mirrors a principle from [Think Slowly, Ship Fast](/blog/think-slowly-ship-fast): decide before you generate, not after. The same principle drove our [990-file monorepo reorganization](/blog/where-does-your-code-belong) — the architecture was already in the imports. The script just read it.

## What The Code Actually Becomes

This is not abstract. Here is a real transformation.

**Before** — a typical Next.js API route:

```typescript
// pages/api/health.ts (Next.js)
export default async function handler(req, res) {
  const deep = req.query.deep === "true";
  const dbStatus = await checkDb();
  res.status(200).json({ status: "ok", db: dbStatus });
}
```

**After** — an explicit Hono handler on Cloudflare Workers:

```typescript
// routes/health.ts (Hono + Cloudflare Workers)
import type { Context } from "hono";
import { Hono } from "hono";
import type { Env } from "../../core-logic/env.js";

const health = new Hono<{ Bindings: Env }>();

health.get("/health", async (c: Context<{ Bindings: Env }>) => {
  const deep = c.req.query("deep") === "true";
  const [r2Status, d1Status] = await Promise.all([
    checkDependencyHealth(() => c.env.R2.head("__health_check__")),
    checkDependencyHealth(() => c.env.DB.prepare("SELECT 1").first()),
  ]);
  return c.json(buildHealthPayload({ r2: r2Status, d1: d1Status }));
});

export { health };
```

No file-based routing convention. Typed bindings via `c.env` instead of environment variables. The handler is an explicit function, not a default export relying on framework magic.

Auth middleware follows the same pattern — explicit, composable, visible:

```typescript
// middleware/auth.ts (Hono)
app.use("/proxy/*", authMiddleware);
app.use("/api/checkout", authMiddleware);
app.use("/api/billing/*", authMiddleware);
```

Every protected route in one place. No `_middleware.ts` files scattered across directories. These are from [our production edge API](https://github.com/spike-land-ai).

The full target stack:

| From | To |
|---|---|
| Next.js App Router / Pages Router | TanStack Router |
| `next.config.js` | `vite.config.ts` |
| Vercel deployment model | Cloudflare Workers |
| Next.js middleware | Hono middleware |
| `getServerSideProps`, ISR | explicit loaders, edge handlers |
| `next/image`, `next/font` | direct imports, platform-native handling |

The real benefit is not "new shiny tool." It is a simpler runtime model, faster local loops, fewer hidden conventions, and less framework rent. The full reasoning: [The Universal Interface Wasn't GraphQL](/blog/the-universal-interface-wasnt-graphql).

## Why Hard Repos Are Still In Scope

Most migrations die against complex codebases for three reasons. And all three are solvable once you name them.

1. **Framework coupling is implicit.** Data loading, auth, routing, rendering, and deploy assumptions — mixed together, invisible, load-bearing.

2. **Business logic is in the wrong layer.** Server logic leaks into UI, UI state leaks into backend assumptions, nobody is sure what is authoritative.

3. **Teams cannot separate mechanical work from judgment.** So everything gets treated like custom engineering, making the project too expensive to attempt.

**Mechanical changes get formalized:** route rewrites, config conversion, import replacement, directory reshaping, environment remapping, deploy config, asset and middleware rewiring.

**Judgment-heavy changes get isolated:** Vercel-only platform assumptions, long-lived Node.js dependencies, custom SSR business logic, ad hoc middleware side effects, undocumented Next.js internals.

The hard parts do not disappear. They stop contaminating the whole migration.

Their structure is encoded in imports, dependencies, boundaries, and runtime assumptions. A script sorted hundreds of our files because the repo already contained the architecture. One operator is not statistical proof of scale. But it is evidence that the method crosses the developer boundary when the structure is made visible enough. The right role for human expertise is not brute-forcing the migration by hand — it is stepping in where deterministic passes hit real business complexity.

## What Does Not Migrate Automatically

False confidence is expensive. So here are the boundaries:

- Deeply Vercel-specific infrastructure behavior
- Code assuming long-lived Node.js APIs or server processes
- Undocumented Next.js internals
- Application logic accidentally encoded inside framework behavior
- Repos with weak test coverage on core paths (read [The Testing Pyramid Is Upside Down](/blog/the-testing-pyramid-is-upside-down) first)

A migration service should reduce risk, not hide it.
