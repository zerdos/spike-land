# spike-chat: Context Compression Architecture

## Problem
Transcript-first chat (append full history each turn) degrades reasoning as prompt grows. Signal/noise ratio drops before context window fills.

## Proof
Single gemini-3-flash agent + 1 PRD rewrote entire frontend to Astro. No errors. No swarm. Context discipline > model capability.

## Architecture: 7 Mechanisms

### 1. Split System Prompt
- Stable prefix: identity, values, operating rules (fixed, cache-friendly)
- Dynamic suffix: user-specific memory, max ~800 tokens, explicitly pruned

### 2. Scored Memory Notes (not transcript)
Each note carries: trigger, lesson, confidence score, help count, recency.
- Below demotion threshold → dropped
- Keeps helping → promoted
- Selection: greedy under fixed budget, sorted by (confidence × recency)
- Filter question: "What is still worth carrying forward?"

### 3. Four-Stage Pipeline (context isolation)
1. **Classify** → structured intent
2. **Plan** → response strategy (receives classify artifact)
3. **Execute** → uses plan artifact + bounded tool surface
4. **Extract** → updates long-term memory (background)

Each stage receives prior stage's artifact, NOT full raw process.

### 4. History Compression Budgets
- Historical memory: ≤8 prior rounds
- Historical token budget: ~8% model window, cap 6,000 tokens
- Stage working memory: ~5% model window, cap 4,000 tokens
- Assistant text summaries: 320 chars
- Tool args summaries: 180 chars
- Tool result summaries: 360 chars

Older turns → compact summaries. Budget exceeded → collapse older summaries.

### 5. Tool Work Compression
Raw tool traces → compact stage artifacts:
- tool name
- target-oriented argument summary
- compact result summary
- browser surface: page title + URL + few target labels + short text preview

No replaying full payloads.

### 6. Bounded Tool Calling
Execute stage only:
- Fetch MCP tool catalog → bound to compact set
- Tool definitions wired into execute-stage model call (not just route-advertised)
- Stream tool_call_start/tool_call_end events to UI

### 7. Completion ≠ Truth
- Tool calls anchored to assertion IDs
- Tool outputs = evidence records
- Assertion state tracked separately from task state
- Rule: tool output is evidence → updates assertion state → conflicts stay unresolved

## Core Principle
Model needs smallest truthful artifact preserving: user intent, relevant durable memory, current plan, unfinished work, evidence from currently-relevant tools.

The classic AI chat failure is not a dumb model.

It is a smart model drowning in its own history.

Every turn, the system appends more conversation, more tool output, more stale context. The signal-to-noise ratio drops. Attention gets diluted. And a model that was sharp at turn 3 starts producing mush by turn 15 — not because it got worse, but because the working set got noisier.

Think of it like a desk. A clean desk with three relevant documents lets you think clearly. The same desk buried under six months of printouts, sticky notes, and old coffee receipts makes even simple tasks harder. The desk did not shrink. The clutter grew.

That is the problem we set out to solve in spike-chat.

## The Proof Point: One Agent Rewrote The Whole Frontend

This is not a theory post.

We migrated the entire site to Astro with a single agent. Not a swarm. Not a human manually chunking the work into fifty tickets. Not a more powerful model with an oversized prompt.

One agent. One PRD. `gemini-3-flash`. Continuous context compression through the stage-memory system. One full frontend rewrite.

And it did not make the usual context-collapse mistakes — the kind where the model forgets a constraint it handled correctly ten turns ago, or contradicts a decision it made an hour earlier.

That is the point of this architecture.

When the context stays sharp, a fast model can do work that transcript-first systems bungle even with an expensive model. If the representation is right, the model spends its attention on actual product logic instead of re-reading a decaying wall of chat history.

The Astro migration is a useful proof point because drift shows up fast in a whole-site rewrite. But the same working-set discipline matters even more in open-ended work, where drift is subtler and harder to catch.

**Better context discipline beats bigger transcript budgets.** That is the commercial message and the technical one.

## Transcript-First Chat Is The Wrong Primitive

Most AI chat systems follow a simple rule:

> Take the entire chat so far. Append the next user message. Send it all back to the model.

This feels natural because it matches the UI. But it creates a predictable degradation curve:

- every turn increases prompt size
- old tool results stay alive long after they stop being useful
- the model re-reads work it already completed
- browser and tool output crowd out the actual task
- reasoning quality drops long before the context window fills up

The transcript becomes the product. But the transcript is not a good working memory — it is a noisy, growing, uncompressed record of everything that happened, including everything that stopped being relevant.

The right abstraction is a compact working set: the minimum state that still preserves what matters. Not the full history. The distilled history.

One scope note before the implementation details: some of the mechanisms below live directly in spike-chat, and some live in the shared chat runtime and CLI layer. The architecture matters more than which package boundary it sits in.

## What We Changed In spike-chat

### 1. We split the system prompt into a stable prefix and a dynamic suffix

In spike-chat's Aether prompt layer, the system prompt is not one giant mutable blob that gets rebuilt every turn.

It is split into two pieces:

- a **stable prefix** that defines identity, values, and operating rules
- a **dynamic suffix** that contains user-specific memory

The stable prefix stays fixed — like the constitution of a country. It does not change based on the conversation. This helps cache reuse and keeps base behavior consistent.

The dynamic suffix is explicitly bounded and pruned:

- stable prefix: persistent operating contract
- dynamic suffix: max ~800 tokens

User-specific memory is not allowed to grow forever just because the conversation did. The suffix has a budget, and it enforces it.

### 2. We treat memory as scored notes, not immortal transcript

spike-chat does not try to remember everything. It remembers reusable lessons.

Each note has:

- a trigger (when is this relevant?)
- a lesson (what should I do?)
- a confidence score
- a help count
- recency data

Notes below the demotion threshold are dropped. Notes that keep helping get promoted. Selection is greedy under a fixed budget and sorted by confidence times recency.

This matters because the memory system answers a harder question than "what happened before?"

It asks:

> What is still worth carrying forward?

That is a fundamentally different filter. Most chats replay everything like a tape recorder. spike-chat packs only the notes that still earn their place — like a traveler who can only carry one bag and has to choose what goes in it every morning.

### 3. We split every request into stages instead of running one long monologue

spike-chat uses a four-stage pipeline:

1. Classify
2. Plan
3. Execute
4. Extract

The value here is context isolation, not orchestration theater.

Each stage receives the artifact from the previous stage, not the entire raw process:

- classify turns the user request into structured intent
- plan turns that intent into a response strategy
- execute uses the plan artifact plus the current tool surface
- extract runs afterward to update long-term memory in the background

```text
user request
  -> classify -> intent artifact
  -> plan     -> response strategy
  -> execute  -> answer + tool evidence
  -> extract  -> durable memory updates

compressed history feeds each stage as bounded working memory,
not as full transcript replay
```

Think of it like a relay race instead of a one-person marathon. Each runner gets a clean baton — the compact output of the previous leg — not the full story of every step the previous runner took. The baton carries the essential state. The noise stays behind.

This prevents a common failure in single-pass systems: the model has to do classification, planning, tool selection, execution, response writing, and memory formation all inside one swollen prompt. One context window should not be doing all of those jobs at once, any more than one employee should be doing accounting, sales, and customer support simultaneously.

### 4. We compress history instead of replaying it

This is the most important sharpness win. It is also the mechanism behind the "continuous context compression" claim from the Astro migration.

In the shared chat runtime, we added explicit stage-memory compression so older turns and completed stages become compact summaries, not dead weight.

The budgets are deliberate:

- historical memory: up to 8 prior rounds
- historical token budget: roughly 8% of the model window, capped at 6,000 tokens
- stage working memory: roughly 5% of the model window, capped at 4,000 tokens
- assistant text summaries: 320 chars
- tool args summaries: 180 chars
- tool result summaries: 360 chars

The system never blindly says "just send all of it." It does the opposite:

- summarize earlier turns
- keep only the recent relevant rounds
- compress completed stage artifacts
- collapse older summaries when the budget is exceeded

This is how you solve "too much context" in practice. Not by hoping for a million-token model. By managing the working set like a stock in a system — tracking what flows in, what flows out, and what the current level is. The budget is the stock. The compression is the outflow. If inflow exceeds outflow, the stock overflows and quality degrades.

### 5. We compress tool work, not just text

Raw tool traces are one of the biggest hidden context leaks in AI products.

A transcript-first system will happily keep replaying:

- raw JSON arguments
- long browser results
- huge DOM-like payloads
- tool outputs that were only useful for one step

This is like keeping every receipt from every store you visited on a road trip, instead of just remembering where you stopped and what you bought. The receipts are evidence, but after you have processed them, the summary is more useful than the originals.

spike-chat converts tool work into compact stage artifacts. Instead of dragging around the full payload forever, it keeps what matters:

- tool name
- target-oriented argument summary
- compact result summary
- a short browser surface description when relevant

For browser work, we compress down to: page title + URL, a few target labels, a short text preview. Not the full DOM. Not the full screenshot. Just enough to remember what happened and why it mattered.

That is the difference between a usable working memory and a junk drawer.

### 6. We keep tool calling bounded

Tool calling has its own version of the context problem. Section 5 is about what happens after a tool runs. This section is about what happens before the model chooses one.

If you dump a huge tool catalog into every model call, you burn context budget before the model even starts reasoning. It is like handing someone a phonebook when they need one number.

Our broader spike-land work already attacked that problem in [Why Your Claude Agent Is Wasting 70% of Its Context Window on Tool Descriptions](/blog/why-your-claude-agent-wastes-context-on-tool-descriptions).

In spike-chat, the execute stage now does the practical version:

- fetch the MCP tool catalog
- bound it to a compact set
- hand that tool surface only to the execute stage
- stream live `tool_call_start` and `tool_call_end` events back to the UI

The fetched MCP tool definitions are now actually passed into the execute-stage model call — the tool-calling surface is wired into the request, not just advertised by the route.

Tool calling is part of the focused execution context instead of a disconnected side concern. The model sees the tools it needs for this step, not every tool it might ever need.

### 7. We are separating completion from truth

There is one more improvement worth calling out, even though it currently lives in the shared chat runtime and CLI layer:

- tool calls can now be anchored to assertion IDs
- tool outputs become evidence records
- assertion state is tracked separately from task state

This changes the contract in a subtle but important way.

Most chat systems implicitly act like:

> Tool ran = task done = truth established

That collapses three different things — evidence, progress, and truth — into one bucket. It is like saying "I saw the bridge, therefore I crossed the river." Seeing and crossing are different events.

The better rule is:

> Tool output is evidence. Evidence updates assertion state. Conflicts stay unresolved.

A successful browser step should update the evidence for "the page rendered correctly," not silently mark the whole task as complete. Evidence accumulates. Truth is a separate judgment.

That same principle is how spike-chat avoids becoming just another smooth-talking transcript machine.

The frontend rewrite was not magic. It was what happens when a model gets:

- one canonical source document
- a focused tool surface
- compact working memory between steps
- continuous context compression instead of transcript sprawl

That is what let one agent stay coherent across a whole-site migration.

## Why spike-chat stays sharp

Here is the simplest way to describe the difference.

| Transcript-first chat | spike-chat |
|---|---|
| Full history is the default | Compressed working set is the default |
| Memory means "keep everything" | Memory means "keep what still earns space" |
| Planning, execution, and memory all happen in one prompt | Work is split into stages |
| Raw tool traces accumulate | Tool work is summarized and bounded |
| Tool catalogs can bloat context | Tool surface is bounded at execution time |
| Old context lingers because deleting it feels risky | Old context is deliberately compacted |

spike-chat stays sharp not because it has infinite context, and not because it uses a magical model. It stays sharp because it is disciplined about what deserves to stay in memory.

That discipline is what let a single `gemini-3-flash` agent take one PRD and rewrite the whole frontend into Astro without drifting off the rails.

## The Bigger Point

The future of good AI chat is not larger transcript windows.

It is better representation.

The strongest claim we can make is not that spike-chat has the most tools or the largest context window. It is that spike-chat is unusually strict about what gets to stay in working memory.

spike-chat is not sharp because it hoards context.

spike-chat is sharp because it refuses to.

The model does not need your entire conversation history. It needs the smallest truthful artifact that preserves:

- the user's actual intent
- the relevant durable memory
- the current plan
- the unfinished work
- the evidence from the tools that matter right now

Everything else is noise. And noise is the enemy of reasoning.

Less replay. More compression. Less noise. More signal.

That is how you keep an AI system sharp.
