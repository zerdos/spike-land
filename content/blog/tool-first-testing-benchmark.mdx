At spike.land, we have a thesis: exposing business logic as typed MCP tools fundamentally changes the developer experience. Faster testing. Cheaper testing. More reliable testing.

But a thesis without numbers is just an opinion with good posture. So we ran the experiment.

We took the concepts from our [A/B Testing Bug Detection](/blog/ai-ab-testing-bug-detection) strategy and put them to the test. We implemented a core piece of business logic — the Bayesian evaluation engine for A/B experiments — as a typed MCP tool (`evaluate_experiment`), then benchmarked it against a conventional browser-level Playwright test doing the same job.

Think of it like this: one approach opens a browser, navigates to a dashboard, clicks buttons, and waits for pixels to appear. The other approach calls the function directly, like picking up the phone instead of driving across town to deliver a message in person.

Here are the raw numbers from our 100-run comparison.

## The Benchmark Setup

**The Test Subject:** The statistical decision rules for A/B testing (Bayesian evaluation / `evaluateExperiment` logic). This logic processes variant impressions and conversions, runs Monte Carlo simulations, and determines if a variant should graduate.

**Approach 1: Browser-Level Testing (Playwright)**
The traditional end-to-end approach: launch Chromium, navigate to the dashboard, trigger an evaluation via the UI, wait for the DOM to update with the "Winner" result. Every layer of the stack is involved — network, rendering, JavaScript execution, DOM mutation.

**Approach 2: Tool-First Testing (Vitest + MCP)**
Invoke the business logic directly via the `evaluate_experiment` MCP tool with the exact same test data. No browser. No network. No pixels. Just the logic and its answer.

## The Raw Numbers

We ran each approach 100 times consecutively:

| Metric | Browser (Playwright) | Tool-First (MCP) | Improvement |
| :--- | :--- | :--- | :--- |
| **Success Rate (100 runs)** | 100% | 100% | - |
| **Total Execution Time** | 26,678.65 ms | 268.51 ms | **99.4x Faster** |
| **Avg Time Per Run** | ~266.79 ms | ~2.69 ms | **99.4x Faster** |
| **Lines of Test Code** | ~20 lines | ~5 lines | **75% Less Code** |
| **Flakiness Risk** | Medium (DOM/Network) | Zero (Deterministic) | **Eliminated** |

*Note: The browser test ran in headless mode without heavy network latency — giving it the best possible baseline. In a real CI environment with cold starts, network variability, and shared runners, browser tests are significantly slower and more prone to flakiness.*

## Why This Matters

The gap is not incremental. It is structural. Two orders of magnitude is not "a bit faster" — it is a different category of thing.

**Speed changes what you can do.** At ~2.7ms per test, you can run thousands of tool-first tests in the time it takes to boot a browser and run a dozen conventional tests. This isn't about impatience. It's about feedback loops. Shorter feedback loops mean faster iteration, which means more experiments, which means better software. The speed of your tests is the speed of your learning.

**Determinism changes what you can trust.** By testing the MCP tool directly, we completely strip away the flakiness introduced by UI rendering, arbitrary timeouts, and network mocking. The test either passes or fails for a real reason — not because a CSS transition took 50ms longer than expected. Flaky tests erode trust in your test suite. Deterministic tests restore it.

**Simplicity changes what you can maintain.** The tool-first test required passing a JSON payload and asserting the result. Five lines. No page objects, no selectors, no wait-for-element chains. Seventy-five percent less code to write, review, debug, and keep working as the UI evolves.

This benchmark turns a platform thesis into hard evidence. By treating MCP tools not as wrappers around existing code, but as the primary interface for core business logic, you achieve a degree of testability that conventional end-to-end testing simply cannot match. The browser test asks: "Does the whole machine produce the right pixels?" The tool-first test asks: "Does the logic produce the right answer?" Both questions matter. But only one of them runs in 2.7 milliseconds.
