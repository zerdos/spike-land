## Le problème dont personne ne parle honnêtement

N fois M. C'est le problème d'intégration caché en pleine vue.

N clients IA. M services externes. Chaque combinaison nécessite de l'authentification, du parsing de réponse, de la gestion d'erreurs, du rate limiting, de la découverte d'outils. Écrit from scratch, à chaque fois.

Si vous avez 5 clients et 10 services, ça fait 50 chemins d'intégration. Ajoutez un service de plus et il vous faut 5 intégrations supplémentaires. La surface croît de manière quadratique. La charge de maintenance suit.

MCP (Model Context Protocol) réduit ce problème N-fois-M à N-plus-M. Un protocole, implémenté une fois de chaque côté. N'importe quel client parle à n'importe quel serveur. C'est la même intuition qui a fait réussir l'USB face à une douzaine de câbles propriétaires — un contrat partagé bat le câblage sur mesure.

C'est le standard sur lequel nous avons construit spike.land. Aujourd'hui, nous lançons.

## Ce qu'est spike.land

spike.land est une **plateforme IA MCP-first** — un registre d'outils alimentés par l'IA que vos agents peuvent appeler via un protocole unique et standardisé. Au lieu de câbler des API individuelles, votre client IA se connecte à spike.land et accède immédiatement à plus de 80 outils prêts pour la production.

Pas d'intégrations custom. Pas de code d'authentification. Pas de logique de pagination. Juste des outils, prêts à être appelés.

### Les 80+ outils que nous livrons aujourd'hui

Notre ensemble d'outils initial couvre cinq catégories :

**Chess Arena** — Des échecs compétitifs complets avec classements ELO, défis en temps réel et gestion de l'état de la partie. Six outils. `chess_new_game`, `chess_send_challenge`, `chess_make_move`, `chess_get_board`, `chess_get_elo`, `chess_resign`.

**Image Studio** — Analyse, génération et amélioration d'images par IA, extraction de palettes, gestion d'albums et orchestration de pipelines. 42 outils construits sur une architecture middleware composable décrite dans [The Architecture of Scale](/blog/the-architecture-of-scale-building-tools).

**Codespace** — Exécution de code en direct, transpilation et environnements de prévisualisation. Cinq outils pour exécuter du TypeScript et du JavaScript dans des sandboxes isolées.

**QA Studio** — Automatisation navigateur via Playwright. Huit outils pour les tests end-to-end scriptés, la capture d'écrans et l'enregistrement d'interactions.

**Utilitaires** — Fetch web, recherche, opérations sur fichiers, et un ensemble croissant d'outils polyvalents qui se composent entre catégories.

Chaque outil suit le même contrat : schéma d'entrée typé en Zod, réponse narrative, sortie JSON structurée. [Pourquoi des réponses narratives ?](/blog/mcp-explained-universal-adapter-for-ai-tools#narrative-responses-tell-stories-not-dump-data) Parce qu'un agent IA n'a pas seulement besoin de données — il a besoin de données sur lesquelles il peut raisonner en contexte.

## Le Marketplace d'outils MCP

Nous lançons le marketplace spike.land en même temps que la plateforme.

L'idée est simple : si vous construisez un outil MCP utile, vous devriez être payé quand d'autres l'utilisent. Nous gérons l'infrastructure — auth, rate limiting, facturation, découverte, conformité au protocole. Vous gérez la logique métier. La plomberie est notre problème, pas le vôtre.

**Partage de revenus : 70/30. Les auteurs d'outils gardent 70%.**

Les outils publiés apparaissent dans le registre spike.land et deviennent appelables par tout utilisateur de spike-cli. La facturation est basée sur les crédits — les utilisateurs paient par appel, vous gagnez par exécution réussie.

Pour publier un outil, vous avez besoin de :
1. Une définition d'outil MCP valide (nom, description, schéma Zod, handler)
2. Un compte développeur spike.land
3. Un package npm publié ou un endpoint Cloudflare Worker déployé

Le [pattern tool builder](/blog/the-architecture-of-scale-building-tools) que nous utilisons en interne est le même que celui utilisé par les auteurs du marketplace. Même surface d'API. Même format de réponse. Facile à découvrir et à composer pour les agents.

Nous ouvrons le marketplace aux premiers partenaires aujourd'hui. Si vous avez des outils à publier, [postulez pour un accès anticipé](https://spike.land/marketplace/apply).

## API Gateway : Crédits et BYOK

Beaucoup de nos outils appellent des modèles IA en interne — analyse d'images, explication de code, génération de contenu. Nous vous donnons deux façons de payer ces appels de modèles, parce que la flexibilité compte quand on cherche ce qui marche.

**Apportez vos propres clés (BYOK)**

Connectez vos clés API OpenAI, Anthropic ou Replicate dans le tableau de bord. Les appels d'outils qui nécessitent un modèle passent par vos clés. Votre coût. Vos limites de débit. Contrôle total.

**Crédits proxy Spike.land**

Vous préférez ne pas gérer plusieurs comptes API ? Utilisez les crédits proxy spike.land. Un pool de crédits, une facturation consolidée, pas de comptes par service.

- 1 crédit = 1 appel API de modèle standard (approximatif, varie selon le modèle et le nombre de tokens)
- Les crédits se reportent mensuellement sur les plans Pro et Business
- Achetez des crédits supplémentaires à 0,10$/crédit

Le proxy est un passthrough — nous n'ajoutons aucune marge sur les coûts de modèle au-delà de la conversion en crédits. Notre marge est sur la plateforme et le marketplace, pas sur l'inférence de modèle.

## La stack technique

spike.land tourne entièrement sur Cloudflare Workers. Pas de serveurs à provisionner. Pas de régions à configurer. Les requêtes sont routées vers le nœud edge le plus proche automatiquement. L'infrastructure s'efface pour que les outils puissent faire le travail.

- **Edge API** : Framework Hono sur Cloudflare Workers
- **Registre MCP** : Cloudflare Workers + D1 (SQLite à l'edge)
- **Auth** : Better Auth avec Drizzle ORM
- **Frontend** : Vite + React + TanStack Router
- **Runtime des outils** : Environnements Worker isolés par catégorie d'outil

L'[architecture complète](/docs/architecture) est documentée. Nous construisons en open et nous continuerons.

## Pourquoi MCP ?

Nous aurions pu construire une API REST. REST est familier. Tout le monde sait comment faire un curl sur un endpoint.

Mais les API REST ont été conçues pour des développeurs humains qui fixent de la documentation. Elles optimisent pour du JSON compact, des codes de statut, des tokens de pagination. Les agents IA ont besoin d'un contrat différent — un qui dit au modèle *ce que* fait une capacité en langage naturel, *comment* l'appeler avec des paramètres typés, et *ce que* la réponse signifie en contexte.

MCP est ce contrat. Il définit la découverte, les schémas typés et les réponses narratives comme des primitives de première classe. Quand votre agent IA se connecte à spike.land via MCP, il n'a pas besoin de pages de documentation. Il lit les descriptions des outils, infère les bons appels et exécute. Le protocole lui-même est la documentation.

Nous en avons écrit davantage dans [MCP Explained](/blog/mcp-explained-universal-adapter-for-ai-tools). La version courte : MCP est une couche de présentation, pas juste un transport. Il façonne la manière dont le modèle pense à vos capacités, pas juste la manière dont il les appelle.
