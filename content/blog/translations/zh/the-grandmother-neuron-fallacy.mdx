## 凌晨两点的 bug，但它不是 bug

我有一个三工具 AI 流水线。工具 A 获取用户资料。工具 B 获取最近活动。工具 C 起草个性化邮件。每个工具都有单元测试。通过了。集成测试也通过了。

然后我端到端运行 agent，大约十次中有一次，它会失败。

不是崩溃。是漂移。

工具 A 返回的用户简介碰巧提到了 `React Native`。LLM 抓住了这个词组，在调用工具 B 时编造了一个移动活动参数。工具 B 返回空数组，因为那个参数没有匹配任何真实内容。工具 C 写了一封关于一个不存在的产品的邮件。

每个工具都正确工作了。但流水线坏了。

我花了三个小时像调试普通软件问题一样调这个：收紧 prompt、添加验证器、锁死 schema。把系统当作一个齿轮链来处理，其中一个齿轮打滑了。

那就是错误所在。这不是齿轮链。这是一种完全不同的系统。

## 谬误

在神经科学中，"祖母神经元"是一个想法：你大脑中某处有一个特定的神经元专门负责一个特定的概念——你的祖母。这是一个诱人的想法，因为它很干净。一个部件，一个功能，一个含义。就像一个组织良好的文件柜。

神经科学几十年前就超越了这个观点。大脑不是这样运作的。概念从分布式网络的模式中涌现，而不是从单一的专用细胞中产生。你的祖母不是存储在一个神经元中的——她是数百万神经元的一个激活模式，而这个模式会根据你是在记忆她的脸、她的声音还是她泡茶的方式而发生变化。

我们在 AI 工程中还没学到这个教训。

我们构建一个叫 `fetch_user_activity` 的工具，然后假装它有一个固定的含义。在隔离状态下确实如此。但一旦这个工具的输出进入 LLM 的上下文窗口，它就不再只是一个函数返回值了。它变成了语义场的一部分——被附近的文本、之前的工具调用、命名选择以及争夺模型注意力的其他一切所塑造。

**错误不在于构建确定性工具。错误在于假设它们在模型开始推理后还能保持确定性。**

## 钟表和云

Karl Popper 有一个有用的区分：钟表和云。钟表是机械的、可预测的。你可以拆开它们，理解每一个齿轮。云是不规则的、依赖上下文的。你可以用统计方式描述它们的行为，但无法预测明天下午 3:47 的确切形状。

软件工程师擅长构建钟表。REST 端点是钟表。数据库事务是钟表。我们知道怎么测试这些，因为它们每次都以相同的方式运行。

LLM 是云。

不是随机的——云有结构、趋势、模式——但也不是机械可预测的。措辞、工具顺序或先前输出的微小变化可以改变它采取的路径。同样的输入在不同运行时可能产生不同的推理，而两次运行可能都是完全合理的。

那我们在这些系统到来时做了什么？我们试图强迫云像钟表一样运行。严格的 JSON。严格的 schema。假设足够的结构会让推理变得确定性。

结构有帮助。但它不能解决核心问题，因为不稳定性不在格式化层。它在推理层。你可以把输出强制成完美的 JSON，但仍然得到错误的答案，因为模型在格式化之前就做了错误的决策。

这就是为什么面向模型的系统不仅需要严格的 schema。它们需要语义清晰度：解释意图而非仅仅是参数的工具描述。保留含义而非仅仅保留结构的响应。

## 变异并不总是噪声

在传统软件中，只有一条正确路径，任何偏差都是 bug。这个假设对钟表有效。

对云就不那么有效了。

当同样的 prompt 在不同运行中产生略有不同的推理时，我们本能地把它当作不可靠。我们降低温度、添加约束、收窄通道。有时候这是对的。

但有时变异在告诉我们什么。系统在导航一个真正复杂的空间，而不是沿着一条固定路线前进。就像一条河在每个春天走的路径略有不同——变异不是河流的失败。它是关于地形的信息。

在工具链中这变得危险。如果你的系统依赖于一个确切的措辞、一个确切的参数名、一个确切的中间状态序列，你就不是构建了一个健壮的 agent。你构建了一条窄通道，然后祈祷模型永远不要蹭到墙壁。

更好的目标不是消除变异。而是构建即使模型轨迹偏移也能保持可读性的接口。

## 工具链中的蝴蝶效应

一个微小的细节出现在一个工具响应中。一个略有不同的措辞进入上下文。突然模型开始进行你在测试中觉得不可能的调用。

这是 LLM 系统中的蝴蝶效应，它和传统 API 故障有本质区别。如果你把错误的 ID 传给 REST 端点，你会得到错误的用户，但故障是局部的，响应格式保持稳定。爆炸半径是一次调用。

在 LLM 中，故障是非局部的。链条早期的一个语义推动可以改变模型在后续每一步关注的内容。就像对话中一句随意的话改变了此后所有内容的基调。

这就是脆弱的工具契约为什么这么容易失败。如果模型必须命中一个确切的键名、一个确切的参数模式、一个确切的解读，一个微小的上下文偏移就会打破一切。

人类可读的工具描述有帮助，因为它们在变异中保留了意图。"请提供数字用户标识符"比一个裸字段名 `uid` 更健壮。模型即使在局部措辞漂移时也能恢复含义，因为描述给了它足够的上下文来自我纠正。

## 增加工具改变一切

这是一个害人的还原主义习惯：

> 我测试了工具 A。我测试了工具 B。所以同时有两个的系统应该可预测地运行。

在 LLM 环境中，添加一个工具不只是添加一种能力。它改变了整个系统的决策面。新的词语出现在上下文中。新的行为变得可想象。注意力转移了。

我见过一个 agent 正确使用 `insert_database_record` 数百次。然后我添加了一个 `search_web` 工具，突然它开始用 URL 而不是清洗后的值填充数据库字段。

网络搜索工具改变了模型认为什么是相关的、可用的、自然要输出的。就像在食谱中加入一种新食材——它不只是在那里待着。它改变了锅里其他所有东西的味道。

多一个工具不是多一个分支。它重塑了每一个决策的概率景观。这就是为什么把几十个工具连接到一个 agent 不像给它一个独立 API 的干净列表。你在改变每一个选择被做出的环境。工具描述需要清晰、有区分度、有语义基础。仅靠裸 JSON schema 无法保持稳定。

## 还原主义仍有其位

这些都不意味着确定性工程无用。它意味着它有一个边界，这个边界就是模型。

数据库写入应该是确定性的。支付流应该是确定性的。你的后端工具仍然应该像钟表一样被测试。这种精确性是必不可少的，LLM 不会改变这一点。

失败发生在你假设钟表的逻辑会不变地穿越模型边界时。它不会。当一个确定性输出进入概率推理器的上下文时，它变成了众多输入之一，受制于解释、注意力竞争和语义漂移。

让你的后端保持精确和结构化。但一旦那种能力暴露给 LLM，面向模型的层就需要考虑语义解读：更清晰的描述、更好的可发现性、工具之间更好的区分度，以及保留意图而非仅保留结构的响应格式。

叙事不替代工程。语义框架帮助确定性系统在概率推理的接触中存活。

## 一句话版本

你的工具是确定性的。你的 agent 不是。相应地设计边界。
