私たちは80以上のMCPツールを半ダースのCloudflare Workersで運用している。先週まで、エラーは`console.error()`に行って虚空に消えていた。ユーザーにはバグを報告する方法がなかった。そしてレート制限はIPごとのフラットなウィンドウ——初回訪問者と数ヶ月プラットフォームを使っているパワーユーザーに同じ制限だった。

三つの別々の問題。一つのシステムがすべてを解決した。

その方法を説明しよう。

---

## 問題：内部DDoS

80のツールをMCP経由でインターネットに公開すると、一人の悪意あるアクターで非常に高額な午後を迎えることになる。リトライループにはまった一つのエージェントがWorkers、D1データベース、KVストアを叩きまくれる——そして外側から見ると、正当なユーザーの高速ツール呼び出しと区別がつかない。

IPベースのレート制限では足りない。エージェントはIPを共有する。VPNも存在する。本当の問題は「1分あたり何リクエストか？」ではない。「この呼び出し元はそもそも信頼すべきか？」だ。

もっとスマートなものが必要だった。壁ではなく、グラデーション——サーモスタットが温度に基づいて調整するように、行動に基づいて自らを調整するシステム。

レピュテーションシステムが必要だった。

## なぜELOか？

コードベースにはすでにELOエンジンがあった——チェスパッケージがプレイヤーランキングに使っている。ELOにはほとんどのランキングシステムにない美しい特性がある：信頼をエンコードする単一の数値で、結果に基づいて継続的に自己調整する。

システム思考では、ELOはバランスフィードバックループを持つストックだ。良い行動がタンクを満たす。悪い行動が消耗させる。誰かが手動で判断を下さなくても、システムが自己修正する。

洞察はこうだ：**ELOはチェスプレイヤーだけのものではない。限られたリソースを争うあらゆるものに使える。**

そこで二つのものにELOレーティングを与えた：
1. **バグ** — どのバグが最も重要か？
2. **ユーザー** — 誰にプラットフォームアクセスを信頼すべきか？

## バグ対バグELO

すべてのバグはELO 1200から始まる。誰かがバグを報告すると、そのバグは同じカテゴリのランダムなバグに対して「勝利」する。そのELOが上がり、相手のELOが下がる。

これが創発的な優先順位付けを生む：
- 頻繁に報告されるバグは自然にリーダーボードを上昇する
- もう誰も遭遇しない古いバグは自然に減衰する
- ELOリーダーボードが自己維持する生きた優先順位リストになる

プロダクトマネージャーが手動でバグをトリアージすることは望まなかった。ユーザーレポートの集合的シグナルを通じて、バグ自身にどれだけ重要かを教えてもらいたかった。優先順位は割り当てられない。創発するのだ。

### バグライフサイクル

```
CANDIDATE → ACTIVE → FIXED → DEPRECATED
```

バグは最初に報告されるとCANDIDATEとして始まる。3つの独立した報告の後、自動的にACTIVEに昇格する。修正されるとFIXEDにマークされる。5セッション以上無関係なら、ELO減衰がDEPRECATEDに向かわせる。ライフサイクルは自動で回る。

## ユーザーELO：数値としての信頼

すべてのユーザー（とそのエージェント）はELO 1200から始まる。アクションがスコアを移動させる：

| イベント | ELO変動 |
|-------|-----------|
| 有効なバグを報告 | +25 |
| 報告したバグが確認される | +10 |
| ツールの成功的な使用 | +1 |
| 誤ったバグ報告 | -15 |
| レート制限にヒット | -5 |
| 悪用フラグ | -50 |

ゲーミング防止のため日次獲得上限+100がある。数学はチェスのK-factorsを参考にしている——新規ユーザーは高いボラティリティ（K=40）で、時間とともに安定する（ELO 2400以上でK=16）。チェスと同じく、初期の対局がより重要だ。

### ティアゲーティング

ELOは三つのティアにマッピングされる：

| ティア | ELO範囲 | 効果 |
|------|-----------|--------|
| Free | 0-999 | 4倍のレート制限 |
| Pro | 1000-1499 | 標準アクセス |
| Elite | 1500+ | フルアクセス、低い制限 |

新規ユーザー（ELO 1200）はProとして始まる。良い行動でそこに留まる。悪用でFreeティアに落ち、レート制限が4倍厳しくなる——実質的に1分あたり120リクエストではなく30リクエスト。

これがアンチDDoSメカニズムだ。意図的に穏やかだ。悪意あるアクターは段階的に悪化するパフォーマンスを経験し、プラットフォームが使い物にならなくなる。正当なユーザーは何も気づかない。ゲートではなくグラデーションだ。そしてハードバンと異なり、回復は常に可能だ——良い行動をすればスコアが回復する。

## 学んだこと

1. **ELOはユニバーサルな信頼プリミティブだ。** 関連性を競うエンティティがあるあらゆるシステムでELOを使える。バグは注目を競う。ユーザーは信頼を競う。数学は同じ——ドメインだけが変わる。

2. **エージェントは人間よりも優れたバグレポーターだ。** 即座に、完全なコンテキスト付きで報告し、手動テストでは見逃すエッジケースをヒットする。システム内の最良のセンサーだ。

3. **レピュテーションベースのレート制限はIPブロッキングよりも穏やかで賢い。** ハードな壁の代わりに、悪意あるアクターはサービスの劣化を経験する。良い行動で回復できる。ゲートではなくグラデーション——グラデーションはゲーミングしにくい。

4. **公開バグトラッキングは信頼を構築する。** ユーザーが自分の報告が修正につながるのを見られれば、より多くのバグを報告する。ELO経由の優先順位を見れば、なぜあるバグが先に修正されるか理解する。透明性が強化ループを作る。

5. **フィードバックツールはどこにでもあるべきだ。** すべてのMCPサーバーにフィードバックツールがあれば、すべてのエージェントインタラクションが潜在的なバグ報告になる。カバレッジは自動的だ。フィードバックを求める必要がない——システムが存在することで収集する。
