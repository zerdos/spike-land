先週、ソフトウェア開発に対する考え方を変える実験をした。

Claude Codeにフィンテックアプリの計画を立てるよう頼んだ。コードを一行も書かせず——計画だけ。そして16の異なるサブエージェントを同時に使ってすべてを調査するよう指示した。

16のエージェントがやったことはこうだ：
- **4エージェント**が認証パターンとAPI設計をマッピング
- **3エージェント**がデータベーススキーマを設計し法的コンプライアンスを調査
- **4エージェント**がモダンなグラスモーフィックインターフェースのUIコンポーネントを選定
- **3エージェント**がアプリを4言語に翻訳する方法を解決
- **2エージェント**がエッジケースとエラーメッセージを文書化

47ファイルの詳細な計画文書を作成した。通常はプロダクトマネージャーとシニアエンジニアのチームが数日かかるような仕事だ。

次にその計画を完全に別のAIモデル——Gemini Flash——に渡して言った：**「これを作れ。」**

**70分後**、アプリが稼働していた。本人確認、滑らかなアニメーション、PIN作成、洗練されたデザイン。すべてだ。

気づいたのは「AIは速い」ということではなかった。気づいたのはこれだ：**計画が十分に良ければ、基本的なモデルでも完璧に実行できる。**

これはシステムの洞察だ。出力の品質はビルダーの知性で決まるのではない。システムに流れ込む指示の品質で決まるのだ。優れた設計図を持つ平凡な建築家は、ナプキンのスケッチしか持たない優秀な建築家を上回る。

## 仕事の新しい形

ジュニア開発者にとって、これは脅威に聞こえるかもしれない。AIがコードを書くなら、自分に何が残る？

ここに秘密がある：AIはプロジェクトを運営しない。あなたがする。変化は「コードを書く」から「エージェントをオーケストレーションする」への移行だ。オーケストラのすべての楽器を演奏することと、指揮することの違いのようなものだ。

現代の開発が実際にどう機能するか：

| フェーズ | 担当 | 理由 |
|-------|------------|-----|
| **計画** | Claude Code | 何かを書く前にコードベースとエッジケースを調査する。 |
| **実装** | Jules（AIエージェント） | 計画に正確に従い、自動テストを追加する。 |
| **CI/CD** | パイプライン | AIが何も壊していないことを確認するためにテストを実行する。 |
| **コードレビュー** | Claude Code | セキュリティと品質について厳密にレビューする。 |
| **修正** | Jules | テストとレビューが通るまでコードを修正し続ける。 |

あなたの新しい仕事は？**何を作りたいかを定義し、それが動作することを検証する。**

フィードバックループに注目してほしい。計画が実装に流れ、実装がテストに流れ、テストが修正にフィードバックし、修正がレビューに戻る。システムが自己修正する。あなたの役割は初期方向を設定し、ドリフトを監視することだ。

## セーフティネット：CI/CDが譲れない理由

「でもAIはハルシネーションする。アプリを壊したら？」と思うかもしれない。

もっともな懸念だ。AIにコードを盲目的にプッシュさせれば、壊れる。だから**CI/CDパイプライン**——継続的インテグレーション／継続的デプロイ——が必要なのだ。

CI/CDは山道のガードレールのようなものだと考えてほしい。ガードレールがあるから道路は速いのであって、遅くなるのではない。ガードレールなしでは這うように進まなければならない。

AIがコードを書くたびに、パイプラインが実行する：
1. **リンティング** — タイポとスタイルエラーのチェック
2. **型チェック** — データ型が合っているか確認
3. **自動テスト** — ロジックが動作するかユニットテストを実行
4. **ビルド** — アプリ全体のコンパイルを試行

すべてのチェックが通らない限り、AIはコードをマージできない。これが自律エージェントを安全にするものだ：信頼ではなく、検証だ。パイプラインはエラーが複合化する前にキャッチするバランスループだ。

## プロジェクトを「エージェント対応」にする

乱雑なコードベースにAIエージェントを投入して魔法を期待することはできない。工場が組立ラインを動かす前にクリーンな入力が必要なように、システムには前提条件がある。

テストに45分かかるなら、AIは待ちぼうけだ。テストがランダムに失敗するなら、AIは幻のバグを追い回し続ける——無駄な労力の強化ループだ。

この速度を実現するには、プロジェクトに必要なのは：

1. **高速なフィードバックループ。** テストは10分以内に実行すべきだ。10秒以内が本当の魔法が起きる場所だ。
2. **厳格なTypeScript。** 型システムは高速道路のレーンマーカーのようなもの。型がエッジの位置を教えてくれるので、AIは車線内に留まる。
3. **堅牢なテスト。** コードが何をすべきかを説明するユニットテスト。実際のユーザーのように振る舞うE2Eテスト。合わせて仕様を形成する。
4. **フレーキーテストゼロ。** テストがランダムに失敗するなら、削除するか修正する。フレーキーテストはテストがないよりも悪い——AIに失敗は普通だと教えてしまう。

自動テストを完全に信頼できるなら、AIにコードを書かせることも信頼できる。信頼は推移的だ。

## 新しい開発者のスキルセット

CI/CDはコードの正しさを担当する。しかし次に何を作るべきかをAIに伝えることはできない。それが新しい仕事だ。

スキルはもう「プロンプトエンジニアリング」——AIを騙す魔法の言葉を見つけること——ではない。**コンテキストエンジニアリング**——AIがあなたの特定のプロジェクトを深く理解し、良い判断を下せるよう適切な背景情報を与えること——だ。

上達すべきことは：

- **成功の定義。** AIがいつ終わったかわかるように明確な受け入れ基準を書く。「完了」を定義できなければ、エージェントもできない。
- **ガードレールの設定。** AIがどのファイルに触れてよく、どのファイルに触れるべきでないか正確に伝える。境界は制約ではない——フォーカスだ。
- **エージェントのデバッグ。** 問題が起きたら、コードだけを読むのではない。AIの推論を読み、なぜ悪い判断をしたか理解し、指示を改善する。プロダクトではなく、プロセスをデバッグしている。

あなたはもうレンガ職人ではない。建築家だ。AIがビジネス判断を必要として行き詰まったときに介入する。もう一つのforループが必要なときではなく。

これをマスターした開発者は10%速くなるのではない。桁違いに速くなる。しかし適応しない開発者はAIに置き換えられるのではない。かつてコンパイラの使い方を学んだように、AIを使う開発者に置き換えられる——使わないことが考えられなくなるほど根本的な力の倍増器として。
