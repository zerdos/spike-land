## The gap between "cool tool" and "useful product"

You can build a good MCP tool in a weekend. Hosting it takes another hour.

Then what?

Think about that gap for a moment. You have something that works. It solves a real problem. But it lives in a GitHub repo where the only people who find it are the ones who already know to look for it.

**How does a useful MCP tool become a product?**

Not a gist. Not a one-off prompt template. A product — something people discover, install, rely on, and eventually pay for.

That missing product layer is what the spike.land app store exists to provide. It is the same gap that the iOS App Store filled for mobile developers in 2008: not the technology to build, but the distribution to be found.

---

## What we mean by "app store"

This is not a storefront of screenshots stapled to landing pages.

On spike.land, an app is:

- a bundle of MCP tools
- a public metadata record
- a discovery surface
- an installable unit
- an execution surface that agents can actually call

The important shift is that last point. The app store is not separate from the runtime. It is the public face of the runtime. When you browse the store, you are looking at live capabilities, not brochure descriptions of capabilities.

Think of it like a farmers market where you can taste everything before you buy it.

---

## Why this matters

The moment you treat MCP apps as first-class products, a reinforcing loop kicks in:

- Discovery becomes searchable and programmable
- Installs become measurable
- Ratings and wishlists become useful signals
- Recommendations can be based on tags, install history, and actual usage
- Developers have a reason to polish, publish, and promote what they build

Each of those makes the next one more powerful. More discovery leads to more installs, which improves recommendations, which drives more discovery.

This is how you get from "AI can call my tool" to "my software has distribution." And distribution is the part that has always been hardest for developer tools.

---

## Open by design

The strongest design decision in the spike.land app store is that it does not trap apps inside the spike.land UI.

You can:

- Browse tool metadata publicly
- Call the MCP runtime from another origin
- Integrate from a Cloudflare Worker
- Package local-first browser apps with IndexedDB

That makes the store more than a website. It makes it a distribution layer for portable app runtimes. Your app lives in the store for discovery, but it can run anywhere — like a book that lives in a library catalog but can be read in any room.

---

## The shared library underneath

The store works because apps share infrastructure instead of rebuilding it.

There are three layers underneath everything:

1. A shared MCP runtime
2. A shared category and discovery layer
3. A shared developer SDK for storage, bundling, and deployment

This is why a new app can feel native to the platform so quickly. The hard parts — auth, discovery, categories, MCP transport, deploy patterns — are already solved. You are not starting from zero every time. You are starting from a working foundation and adding your specific logic on top.

It is the same pattern that made building an iOS app easier than building a desktop app from scratch. The platform absorbs the infrastructure cost, and the developer focuses on what makes their app unique.

---

## Publishing is the new distribution

The commercial model is straightforward:

- Publish useful MCP-native apps
- Let users discover and install them
- Let agents call them from the same runtime
- Reward developers through marketplace economics

The 70/30 split matters, but the deeper value is distribution. Every developer building AI tools faces the same question: "Where does this live? How do users find it?" The app store is our answer.

The tools were always the easy part. Building something useful in a weekend is achievable. Getting it in front of the people who need it — that is the problem nobody has solved for the MCP ecosystem.

Until now.

---

## What comes next

The next stage is about making the store feel inevitable:

- Open submissions with clear review gates
- Stronger public app pages
- Better recommendation quality
- Shared SDK documentation
- More portable deployment paths, including offline bundles

The thesis is simple: if MCP is becoming a real software layer, it needs a real app store. Not eventually. Now.

spike.land is building that layer. The tools were always the easy part. Distribution is the hard part. That is what we are solving.
