# How to Not Produce AI Slop: A Guide for Junior Developers

![A clear, well-structured blueprint of a software architecture standing out against a background of messy, tangled code representing 'AI slop'.](https://placehold.co/800x400)

_By Zoltan Erdos | Brighton, UK | January 2026_

It is a rite of passage: you join a new team, you use AI to help you write a feature, and you proudly open your first pull request. The code looks clean, the tests pass, and you feel great.

Then, the senior developers review it, and the comments pour in. The code uses old patterns. It breaks edge cases. It assumes the database works in a way it hasn't worked since 2023.

You've just produced "AI slop."

Don't panic. It happens to all of us. AI slop isn't syntactically broken code; it's code built on broken assumptions. It looks right, but it doesn't fit the project.

Here is a practical framework to ensure you use AI to produce high-quality, reliable code.

## The Most Important Change: The Planning Interview

The biggest trap is assuming the AI understands your specific codebase just because it can read the files. It sees patterns and makes guesses. When those guesses are wrong, you get slop.

To fix this, flip the dynamic: **During planning, let the AI interview you.**

![A workflow diagram showing the 'Planning Interview' phase where an AI assistant asks questions to a developer before writing any code.](https://placehold.co/600x300)

Before any code is generated, prompt the AI to ask you questions about the problem until it has a complete picture.

*   "What is the exact user flow here?"
*   "What data currently exists on the server?"
*   "What should happen if this specific third-party API fails?"

If you, as a junior developer, cannot answer one of these questions, **stop**. Do not let the AI write code. Go read the documentation, investigate the codebase, or ask a teammate. This process forces you to deeply understand the problem and catches bad assumptions early.

## Create Your Safety Net: Documentation

The AI doesn't know what your tech lead mentioned in yesterday's standup. You have to provide that context.

Before diving into a complex task, gather all relevant information—API docs, old PRs, Slack threads. Use AI tools to summarize this into a personal knowledge base.

When the coding AI suggests an approach, verify it against your notes. Your documentation is your safety net against the AI's lack of project history.

## The Defense Mechanism: Testing

Writing tests before code used to feel like a tedious exercise. Today, it is the only way to prove you understand a problem before asking an AI to solve it.

![A visual representation of Agent-Based Testing, showing a robotic tester interacting with a web UI.](https://placehold.co/600x300)

Think about it: to write a test, you must know what the code should do, understand the inputs and outputs, and think about edge cases. If you cannot write the test, you shouldn't ask the AI to write the solution.

*   **Unit Tests:** Prove that individual functions work.
*   **End-to-End Tests:** Prove the features integrate properly.
*   **Agent-Based Tests:** Let an AI agent log in as a real user and verify the UI behaves correctly.

My rule is simple: **No pull request goes up without tests. Period.** It is the ultimate defense against AI slop.

## TypeScript as Your Guardrail

Strict TypeScript is your first line of defense. The AI might assume an array will never be empty, or a string will always be a valid UUID. With loose typing, this code runs and fails in production.

With strict TypeScript, the compiler stops the slop immediately. It says, "No, this field could be undefined. Handle that case." The AI gets caught before the code review. Enable strict mode and use it aggressively. It catches what you and the AI miss.

## The Checklist Before You Submit That PR

Here is what I check now, every single time:

**Before I ask the AI for help:**
- [ ] Can I explain the problem in my own words?
- [ ] Has the AI interviewed me about the requirements?
- [ ] Do I understand why the current code exists?
- [ ] Have I checked my documentation for relevant context?

**After the AI writes code:**
- [ ] Can I explain every line to a teammate?
- [ ] Have I verified the AI's assumptions?
- [ ] Do I know why the AI chose this approach?

**Before I create the PR:**
- [ ] Do my unit tests prove the code works?
- [ ] Do my end-to-end tests prove the feature works?
- [ ] Does TypeScript pass with no errors in strict mode?

AI coding tools are incredible, but they are tools, not magic. They need your understanding to produce quality work. Master this framework, and your PRs will be your work, enhanced by AI, not slop generated by it.
